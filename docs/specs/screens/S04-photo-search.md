# S4 — 사진 검색 플로우 (08·09·10)

> 입력: `docs/mockups/08-photo-select.html` · `09-analyzing.html` · `10-result.html`,
> `docs/specs/_context/session-context.md`(잠긴 결정·제약), `CLAUDE.md`(루트).
> 설계 원칙: 화면/UX/네비/API 형태는 백지에서 이상적으로. DB·인프라는 후속 세션에서 reconcile.
> 무채색(잉크/그레이, 로즈 금지).

## 스코프

사진 한 장의 분위기로 닮은 관광지를 찾는 3화면 플로우. 사진 선택(08) → 분석 중(09)
→ 결과(10) → (카드 탭) 스팟 상세(07, S3 재사용). 잠긴 결정: **유사도 + 거리순 정렬,
GPS 동의 시에만 거리/거리순칩 활성**.

---

## 공통 결정

### C1. 네비게이션 모델 — push/modal 스택 (탭 아님)

- 사진 검색은 독립 탭이 **아니다.** 현재 화면(홈 등 진입점) 위에 덮였다 닫히는
  **push/modal 스택**. 진입점 위치(홈 히어로/버튼 등)는 S2 홈 세션에서 확정 — 여기선
  "홈 등에서 진입"으로 둔다.
- 스택 순서: `08 → 09 → 10 → 07(스팟 상세)`.
- `←`(back)은 스택을 한 단계 역행한다. 플로우 완료 또는 닫기 시 **진입했던 원래
  화면으로 복귀**.
- 성공 분석 후 `09 → 10`은 **교체(replace)**: 10에서 back은 09를 건너뛰고 08로 간다.
- 목업 3화면 모두 탭바 없이 `←` 버튼만 있는 것과 일치.

### C2. KTO 컴플라이언스 — 업로드 바이트 인메모리 폐기 (불변 규칙)

선택이 아니라 **확정 불변 규칙(invariant).** 설계·구현·리뷰에서 절대 위반 금지.

- 선택한 사진은 분석 시점에 multipart로 **1회 업로드**된다.
- 백엔드는 CLIP 추론을 **메모리에서만** 수행하고 바이트를 **즉시 폐기**한다:
  DB 저장 금지, 디스크 저장 금지, 로그/트레이스에 바이트·base64 기록 금지,
  외부 전송 금지.
- 응답에 사용자 원본 이미지 URL을 담지 않는다(결과 히어로는 클라이언트 로컬 이미지 사용).
- 클라이언트도 선택 이미지를 **앱 메모리(플로우 상태)에만** 보관 — `AsyncStorage`/
  `secure-store`/캐시 파일로 영구 저장하지 않는다. 플로우 종료 시 참조 해제.
- 화면에 별도 "저장 안 함" 안내 문구는 두지 않는다(목업대로 깨끗하게). 저장 안 함
  고지는 **16 약관/개인정보 처리방침 + 본 스펙**에만 명시한다.

### C3. 빈/에러/로딩 = 설계 추가 상태 (목업 happy-path 외)

- 목업 08·09·10은 **정상(happy-path) 한 컷**만 그려져 있다. 08 빈 상태("사진을 고르세요"),
  09 에러("분석하지 못했어요"), 10 빈 결과("닮은 장소를 찾지 못했어요")는 **목업에 없는
  설계 추가 상태**다(honest-minimal 원칙). 스펙 본문의 "목업대로/목업 일치"는 *정상 상태*에
  한함 — 빈/에러 카피는 목업 verbatim이 아니라 본 스펙이 신설한 카피다.

## 08 — 사진 선택 (`08-photo-select.html`)

### 목적
분석할 사진 **1장**을 확보하고 분석을 시작한다.

### 구성요소
- 상단: `←` back 버튼.
- 타이틀 "사진 속 분위기로 여행지를 찾아요" / 서브 "마음에 드는 사진 한 장이면 충분해요."
- 프리뷰 영역(flex로 늘어남):
  - 빈 상태: 점선/일러스트 플레이스홀더 + "사진을 고르세요" 안내.
  - 선택됨: 고른 사진 cover + 우상단 X(remove) 컨트롤.
- 입력 버튼 행: `촬영`(카메라 아이콘) · `갤러리`(이미지 아이콘) — secondary(inset) 2버튼.
- CTA: `분석하기`(스파클 아이콘) — 검정 ink. 사진 없을 땐 회색 비활성.

### 상태
| 상태 | 표현 |
|---|---|
| empty(초기) | 점선 플레이스홀더 "사진을 고르세요", CTA 회색 비활성 |
| selected(normal) | 프리뷰 + X, CTA 검정 활성 |
| (loading 없음) | 분석은 09로 화면 전환하므로 08엔 로딩 상태 없음 |

### 데이터 needs
- 외부 fetch 없음(순수 입력 화면). 출력: 선택 이미지(로컬 URI/바이트) + 분석 트리거.

### 진입/이탈 네비게이션
- 진입: 홈 등 진입점에서 push/modal로 08 진입(빈 상태).
- 이탈:
  - `←` / 닫기 → 진입 화면 복귀.
  - `분석하기` → 09로 push, 동시에 `POST /taste/photo-search` 호출(아래 인터랙션).

### 인터랙션
- **갤러리**: 시스템 사진 선택창(iOS PHPicker / Android Photo Picker)을 띄운다 →
  **권한 팝업 없음**, 고른 1장만 앱으로 전달. 선택 완료 → 프리뷰 상태.
- **촬영**: 탭 시 카메라 권한을 **just-in-time** 요청. 허용 → 촬영 → 프리뷰 상태.
  거부 → "설정에서 카메라 권한을 켜 주세요" 안내(설정 앱 딥링크 제공).
- **X(remove)**: 선택 해제 → empty 상태, CTA 회색.
- **분석하기**: (사진 있을 때만 활성) last-known 위치가 **이미 허용되어 있으면**
  좌표를 조용히 첨부(아래 C-위치). 요청 발사 + 09로 전환.
- 새 사진을 다시 고르면 기존 선택을 교체(단일 슬롯).

### 위치 첨부 (lat/lng 타이밍) — 결정
- 흐름 중 **위치 권한 팝업을 절대 띄우지 않는다.**
- `분석하기` 시점에 위치 권한이 이미 허용(온보딩 04/지도 탭 등에서)되어 있으면
  **last-known/최근 위치**(빠름, fresh GPS fix 대기로 분석을 막지 않음)를 읽어
  요청에 `lat`,`lng`로 첨부 → 결과에 거리 포함, 10에서 거리/거리순칩 활성.
- 허용 안 됨/좌표 없음 → 좌표 생략하고 요청 → 결과에 거리 없음, 10에서 거리 미표시·
  정렬칩 숨김.

### 빈/에러 구분
- empty = "아직 사진 안 고름"(정상 시작 상태). 에러 아님.
- 카메라 권한 거부 = 에러성 안내(설정 유도). 갤러리 시스템창 취소 = 무동작(상태 유지).

## 09 — 분석 중 (`09-analyzing.html`)

### 목적
분석 대기 화면 + 취소/실패 출구 제공. (성공 시 자동으로 10으로 이동.)

### 구성요소
- 상단: `←` back 버튼.
- 중앙: 선택 사진 썸네일(92px) · 타이틀 "사진을 분석하고 있어요" · 서브
  "잠시만 기다려 주세요" · **indeterminate 진행 바**(좌우로 흐르는 막대, 진행률 숫자 없음).

### 상태
| 상태 | 표현 |
|---|---|
| loading | 썸네일 + indeterminate 바. 최소 표시 ~600ms(깜빡임 방지) |
| error | 바 자리에 인라인 오류: "분석하지 못했어요" + "잠시 후 다시 시도해 주세요" + [다시 시도] / [돌아가기] (썸네일 유지) |

> determinate(%) 진행은 쓰지 않는다 — 분석 소요가 가변적이고 단계 정보가 없어
> indeterminate가 정직하다(목업 확정).

### 데이터 needs
- `POST /taste/photo-search` (multipart `image`; optional `lat`,`lng`)의 응답을 대기.
- 응답 → 10으로 전달(결과 페이로드 + queryHadLocation 플래그).

### 진입/이탈 네비게이션
- 진입: 08 `분석하기`에서 push(요청은 08에서 이미 발사, 09는 대기 화면).
- 이탈:
  - 성공 → 10으로 **교체**(back 시 09 건너뜀 → 08).
  - `←`(취소) → in-flight 요청 **중단(abort)** 후 08 복귀(선택 사진 유지).
  - error 인라인 → [다시 시도](같은 사진으로 재요청, 09 유지) / [돌아가기](08 복귀).

### 인터랙션
- `←`: 요청 abort + 08 복귀(사진 유지, 즉시 재시도 가능).
- [다시 시도]: 동일 이미지(+위치)로 `/taste/photo-search` 재요청, loading 복귀.
- [돌아가기]: 08으로(사진 유지).

### 빈/에러 구분 — 중요
- **0건 매칭은 실패가 아니라 성공(빈 결과)** → 09를 통과해 **10의 empty 상태**로 간다.
- 09의 error는 **진짜 오류만**: 네트워크 끊김·타임아웃·서버 5xx·요청 실패.
  - 클라이언트는 `err.code`로 분기(메시지 문자열 분기 금지). 잘못된 이미지/형식 등
    400류도 인라인 오류로 처리하되 카피는 "분석하지 못했어요"로 통일.

## 10 — 결과 (`10-result.html`)

### 목적
내 사진과 닮은 장소를 유사도/거리로 탐색하고 스팟 상세로 진입한다.

### 구성요소
- 히어로(312px): **내가 고른 사진**(클라이언트 로컬 이미지) + 스크림 그라데이션.
  상단 `←` back, eyebrow "내 사진과 닮은", 타이틀 "비슷한 장소 N곳"(N 동적).
- 정렬칩 줄: `유사도순`(기본 on) / `거리순`(off) pill. **GPS 있을 때만 노출.**
- 카드 리스트: 카드마다 이미지(KTO URL) + 글래스 바(이름 / "유형 · 지역 [· 거리]") +
  유사도 게이지(원형, %).

### 상태
| 상태 | 표현 |
|---|---|
| normal | 히어로 + 정렬칩(GPS시) + 카드 리스트 |
| empty(0건) | 히어로 유지(내 사진), 본문에 "닮은 장소를 찾지 못했어요" / "다른 사진으로 다시 시도해 보세요" + [다른 사진으로]→08. 정렬칩 없음 |
| 무GPS 변형 | 정렬칩 줄 **숨김**(유사도순 고정), 카드 메타에서 **거리 텍스트 생략**("유형 · 지역") |
| (loading 없음) | 진입 시점에 데이터 확보됨(09에서 대기 완료). 별도 스켈레톤 불필요 |

### 표기 규칙
- **유사도%** = `round(cosine_similarity × 100)`, 0~100. 게이지 채움 = 그 값. 항상 표시.
  - **정정(S11 §2)**: 원시 `round(cosine×100)%`를 그대로 노출하면 코사인 분포 특성상
    훌륭한 매치도 60~75%로 읽혀 사용자가 "별로 안 닮음"으로 오해한다. → 숫자% 대신
    **버킷 라벨**("매우 닮음/닮음/비슷함")로 표시하거나, 기대 매치밴드를 0~100으로
    **스트레치**해 게이지에 반영한다. 버킷 경계·스트레치 매핑은 D3의 캘리브레이션 결과에
    맞춰 디자인 오너가 확정(확인 항목). 게이지 시각 자체는 유지.
- **거리 포맷(단일 규칙, 공유 util `formatDistance(m)`)**: `<1km → {정수}m`("320m") /
  `1–10km → 소수1 km`("3.4km"·"6.1km") / `≥10km → 정수 km`("18km"·"25km"). 지도(S5)와 **동일 함수**
  (목업 11=`m`, 목업 10=`km` 케이스를 한 규칙으로 통합).
- **유형/지역**: 세분 라벨 · 시도/시군구 라벨(예: "해변 · 제주 제주시").

### 정렬 (결정)
- 정렬은 **클라이언트 재정렬** — 서버 재요청 없음(백엔드가 유사도·거리 둘 다 내려줌).
- 기본 = 유사도순(내림차순). 거리순 = 거리 오름차순(동률 시 유사도 내림차순 tiebreak).
- 무GPS면 거리 데이터가 없어 정렬칩 줄 자체를 숨긴다 — 죽은(disabled) 칩을 남기지
  않는다(honest-minimal). 유사도순 고정.

### 임계값 / 결과 수 (결정)
- 서버가 **유사도 임계값**(구현 시 ~0.60 코사인 튜닝) 이상만, **최대 N개**(~30) 반환.
- 임계값 통과 결과가 **0개면 empty 상태**(위 표). 데이터 없는 극단 케이스도 empty.
- **정정(S11 §2) — 고정 하드컷 → 캘리브레이션 + 소프트 플로어**: "~0.60 튜닝" 헷지를
  실행으로 구체화한다.
  - **캘리브레이션**: 라벨드 쿼리/스팟 쌍 ~100~200개로 FP/FN 교차점을 측정해 임계값을
    정한다(0.50~0.70 예상). 임의 고정 0.60 하드컷 금지.
  - **top-N 소프트 플로어**: 항상 best-K를 노출하고, 플로어 미만 결과는 잘라내지 말고
    **dim 처리**(흐리게)해 보여 준다. 코사인 분포가 쿼리마다 달라 하드컷만 쓰면 좋은
    매치까지 0건으로 떨어뜨릴 수 있으므로, "best는 항상 보이되 약한 매치는 시각적으로
    덜 강조"한다.
  - empty(0건) 판정은 best-K도 의미 있게 못 채우는 진짜 빈 경우로 좁힌다.

### 데이터 needs
```
POST /taste/photo-search
  multipart: image (required)
  query|body: lat, lng (optional — 권한 있을 때만)
→ {
    matches: [ <스팟 카드> + {
      similarity,          // 0~1 (클라가 ×100 반올림)
      distance?            // m, lat/lng 줬을 때만 (SimilarNeighbor.distance와 동일 명)
    } ],
    queryHadLocation: bool // 거리/정렬칩 노출 판단
  }
```
- **`<스팟 카드>` = canonical `{ contentId, title, firstImageUrl, category, regionName, sigunguName }`**
  (백엔드 `SpotCard`/`SimilarNeighbor` 형과 일치). ~~`spotId·name·thumbnailUrl·distanceM`~~ 폐기.
- 카드 메타 "유형 · 지역" = `category`(세분 라벨) · `"{regionName} {sigunguName}"`.
- 카드 탭 시 상세는 **스팟 상세 계약(07, S3)** 재사용 — `contentId`로 조회.
- *reconcile: 현 `/taste/photo-search`는 `PhotoSearchResult{sessionId, detectedMoods, topSpots}`
  (무드 검출형)이라 위 이상형과 다름 → 엔드포인트 재설계 매핑은 S9/S10.*

#### 임베딩 커버리지 / ANN 쿼리 주의 (S11 §2)
- **커버리지 릴리스 게이트**: 임베딩 없는 스팟은 ANN(`embedding <=>`)에 영원히 뜨지 않아
  결과에서 **조용히 실종**된다. 임베딩 ~36% 공백(현 커버리지 ~64%)을 그대로 두고 출시하면
  사진검색 품질이 무작위로 누락된다. → **백필 ~100%를 사진검색 릴리스 게이트**로 둔다.
  운영 중 **커버리지%·빈결과율**을 모니터링한다.
- **graceful degradation**: GPS가 있을 때 ANN 결과가 thin/공백이면, **메타 기반(거리 +
  카테고리)** 보조 결과로 보강해 빈손을 피한다(사진검색 본질은 유지, 어디까지나 보조).
  GPS 없으면 이 보조는 불가 → empty 상태로 정직하게 처리.
- **HNSW 쿼리 형태**: `ORDER BY embedding <=> $1::halfvec(512) LIMIT 30`을 **임베딩 베이스
  테이블에 직접** 걸고, 나온 id로만 메타를 조인한다. 거리식을 메타 테이블과 **cross-join**해
  계산하면 HNSW 인덱스를 못 타니 금지. `EXPLAIN`에 **Index Scan** 뜨는지 확인. 필터를
  추가할 경우 **pgvector ≥0.8 + `hnsw.iterative_scan`** 확인(recall 저하 방지).
